Verteilte Systeme

  • stellt ein anpassbares, flexibles Architekturmuster der Informationstechnik innerhalb des Segments der verteilten Systeme dar
  • Modularisierung von Software (einfach, verständlich, wartbar)
  • unabhängige Entwicklung/Ausführung
  • Kohäsion: ein spezifischer Aufgabenbereich wird erfüllt
  • Koppelung: Module tauschen untereinander Nachrichten aus (niedrige: Module sind unabhängig)

Mermale / Motivation

  • Parallele Verarbeitung (Benutzer und Serviceanfrage)

  • Shared Access: Gemeinsame Nutzung (von Betriebsmittel)

  • Modularisierung: Wiederverwendbarkeit

  • Replikation: Speicherung von Daten auf mehrere Knoten -> Ausfallsicherheit

  • Fehlertoleranz: mögliche Ausfälle müssen berücksichtigt werden

  • Verfügbarkeit: durch Fehlertoleranz erhöht

  • Konsistenz (Daten müssen konsistent bleiben)

  • Heterogenität: für Kommunikation kann unterschiedliche Hard- Software verwendet werden

  • Parallelität: Reihenfolge von Aktionen nicht deterministisch

    • -> synchronisation: Teilaufgaben in der Richtigenreihenfolge agbearbeitet -> gleiches Ergebnis bei gleichem input

  • Gemeinsame Ressourcen: Ein Programmsystem benötigt physikalische und logische Ressourcen

Eigenschaften

Transparenz

  • Tatsache verbergen, dass Prozesse und Ressourcen physisch über mehrere Computer verteilt sind
  • Zugriffstransparenz (Art des Zugriffs auf die Ressourcen verbergen)
  • Positionstransparenz (Ressourcen können beliebig verschoben werden)
  • Ortstransparenz (physisch egal, logischer Name zugeordnet, Positiontransparenz wird vorausgesetzt)
  • Replikationstransaperenz (verbirgt mehrere Kopien)
  • Nebenläufigkeitstransparenz (unabhängige Benutzer ihre Dateien auf demselben Dateiserver abgelegt haben oder auf dieselben Tabellen einer Datenbank zugreifen)

Offenheit

  • einfach erweiter- und veränderbar
  • Interoperabilität (wie sehr Komponenten nebeneinander existieren können)
  • Portabilität (Portabilität beschreibt, in welchem Ausmaß eine Applikation, die für ein bestimmtes verteiltes System entwickelt wurde, ohne Veränderung auf einem anderen verteilten System ausgeführt werden kann.)
  • Erweiterbarkeit (ohne veränderungen anderer Komponenten)

Skalierbarkeit

  • Performance: Geschwindigkeit einer bearbeitenden Anfrage
  • Skalierbarkeit: Geschw bei steigender Last beibehalten (wenn skalierbar, heißt nicht unbedingt auch gute performance (vor allem im single modus))
    • Vertikale Skalierung: Mehr Ressourcen (CPU)
    • Horizontale Skalierung: Mehrere Einheiten (Rechner)

Evolution

Monolithen

  • für einfache Anwendungen
  • keine expliziete Gliederung in Teilsysteme
  • Vorteile:
    • einfache Entwicklung
    • optimierte IDEs
    • einfachere Testung
  • Nachteile:
    • Weiterentwicklung
    • zunehmende Codekomplexität bei Erweiterung
    • Zuverlässigkeit
    • agile Entwicklung erschwert

SOA - Service Oriented Architecture

  • vielzahl von Services kommunizieren miteinander
  • Service:
    • 'Selfcontained System' -> kann autonom ausgeführt werden
    • klare Grenze
    • Zugriff über Schnittstelle
  • Vorteile:
    • Agile Entwicklung
    • Wartbar
    • Offen
    • Isoliert
    • Robust
    • Skalierbar
  • Nachteile:
    • Großer anfänglicher Aufwand
    • Teilweise schlechtere Performance
    • Schwierigerere Konzeption/Planung

HTTP

Kommunikationsprotokolle

  • agreement how the data transmission between several parties take place

forms of syncronisation

  • one way (input to serverice)
  • req, res
  • solicit response (service send client data, client response)
  • notification (service send client data)

Http Basics

  • simple, extensible, stateless, but not sessionless
  • hypertext transfer protocol
  • application level protocol

HTTP Request

[HTTPMethode] [Path] [VERSION]

GET / HTTP/1.1
Accept-Language: en

HTTP Response

[VERSION] [STATUSCODE] [STATUSMESSAGE]

HTTP/1.1 200 OK

Methods

  • GET to get an existing ressource
  • POST to create a new ressource
  • PUT to update an existing ressource
  • DELETE to remove a ressource

status codes

  • 100 codes: Informational requests
  • 200 codes: Successful requests
    • 200 Success
  • 300 codes: Redirects
    • 301 Permanent Redirect
    • 302 Temporary Redirect
    • 304 Not Modified
  • 400 codes: Client errors
    • 401 Unauthorized
    • 403 Forbidden
    • 404 Not found
    • 405 Method not allowed
  • 500 codes: Server errors
    • 501 Not implemented
    • 502 Bad Gateway
    • 503 Service unavailable
    • 504 Gateway timeout

Serviceschicht

ARessource:

@Slf4j  // logger
public class ARessource<T extends AEntity, K extends Serializable> {

    protected JpaRepository<T, K> repository;

    public ARessource(JpaRepository<T, K> repository) { 
        this.repository = repository;
    }

    @GetMapping(path = "/{id}", produces = MediaType.APPLICATION_JSON_VALUE)
    @ResponseStatus(HttpStatus.OK)
    public T read(@PathVariable("id") K id) {
        T entity = repository.getById(id);
        log.info("Read Entity with id: " + id.toString());
        return (T) Hibernate.unproxy(entity);
    }

    @PostMapping(produces = MediaType.APPLICATION_JSON_VALUE)
    @ResponseStatus(HttpStatus.CREATED)
    public T create(@RequestBody T entity) {
        T createdEntity = repository.save(entity);
        log.info("Created Entity with id: " + entity.getId());
        return (T) Hibernate.unproxy(entity);
    }

    @PutMapping(path = "/{id}")
    @ResponseStatus(HttpStatus.ACCEPTED)
    public void update(@PathVariable("id") K id, @RequestBody T entity) {
        repository.save(entity);
        log.info("Updated Entity with id: " + id);
    }

    @DeleteMapping(path = "/{id}")
    @ResponseStatus(HttpStatus.OK)
    public void delete(@PathVariable("id") K id) {
        repository.delete(repository.getById(id));
        log.info("Removed Entity with id: " + id);
    }
}
